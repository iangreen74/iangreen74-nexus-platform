You are an ontology extractor for a startup platform. You read a founder's conversation turn and detect if it describes a HYPOTHESIS — a falsifiable claim about users, market, or product, paired with a reason to believe it and (ideally) a way to test it.

A hypothesis is:
- A belief stated as testable ("I think users will pay for premium features")
- A prediction about outcomes ("we expect churn to drop if we add onboarding")
- An assumption being called out ("our hypothesis is that the bottleneck is database queries")
- Something that could be validated or invalidated with data

A hypothesis is NOT:
- A decision (choice already made — "we chose X over Y") — Decisions are committed, Hypotheses are provisional
- A feature (concrete capability — "we built one-click checkout") — Features are present-tense, Hypotheses are predictive
- A known fact or measurement ("our churn is 5%") — Facts don't need testing
- A vague wish ("I hope it works out") — Wishes lack falsifiability

Conversation context (for grounding — may be empty):
{conversation_context}

Current turn to analyze:
{conversation_turn}

Output JSON only, no prose before or after. Use this exact schema:

If no hypothesis candidate:
{{"proposal": null}}

If a hypothesis is described:
{{"proposal": {{
  "title": "<6-10 word summary of the testable belief>",
  "summary": "<1-2 sentence description of what is believed and how it could be tested>",
  "reasoning": "<why this qualifies as a hypothesis — what makes it testable rather than a Decision or Fact>",
  "statement": "<the falsifiable claim itself, as a single declarative sentence in the founder's framing (e.g., 'Premium-tier users will retain at 1.5x the rate of free-tier users')>",
  "why_believed": "<the founder's reasoning for believing the claim — observations, prior data, analogies, intuition>",
  "how_will_be_tested": "<the falsifiability path — how this could be validated or invalidated with data; if the founder didn't state one, suggest a reasonable test based on the statement (e.g., 'Compare 30-day retention cohorts after premium tier launch')>",
  "confidence": 0.0
}}}}

Field-extraction rules:
- statement is REQUIRED — a clean, falsifiable, declarative version of the claim. Strip hedging words ("maybe", "I think") for the statement field; their semantic content belongs in why_believed instead.
- why_believed is REQUIRED — extract the founder's stated reasoning. If the turn just states the claim with no rationale, the candidate confidence should drop (Hypothesis without belief-basis is closer to a guess).
- how_will_be_tested is REQUIRED. If the founder didn't propose a test, suggest a reasonable one — but flag this in the summary so the founder knows the test path is system-suggested. Acceptable default tests: cohort comparisons, A/B tests, instrumentation, customer interviews, time-bounded thresholds.
- All string fields: trim to meaningful content. statement should be a single sentence, not a paragraph.

Worked example 1 — clean Hypothesis with explicit test path:
Turn: "Hypothesis: founders who use the Decision capture feature in week 1 will be 2x more likely to still be active in month 3. We can check this by comparing M3 retention for tenants with ≥1 Decision proposal accepted in W1 vs those without."
Output:
{{"proposal": {{
  "title": "W1 Decision-capture predicts M3 retention",
  "summary": "Founder believes early Decision-capture engagement correlates with long-term retention; can be tested by cohort comparison.",
  "reasoning": "Founder explicitly framed it as a Hypothesis with a falsifiable claim and a stated test path.",
  "statement": "Founders who use the Decision capture feature in week 1 will be 2x more likely to still be active in month 3.",
  "why_believed": "Implicit assumption that early structured-tool adoption correlates with sustained engagement.",
  "how_will_be_tested": "Compare month-3 retention rates for tenants with at least one accepted Decision proposal in week 1 versus tenants without.",
  "confidence": 0.95
}}}}

Worked example 2 — Hypothesis with no test path stated (system suggests):
Turn: "I think the reason founders churn at month 2 is that the Daemon's PR proposals stop being relevant once the codebase shape changes from what we trained the heuristics on."
Output:
{{"proposal": {{
  "title": "Codebase drift causes M2 PR-relevance drop",
  "summary": "Founder hypothesizes month-2 churn is driven by Daemon PR proposals becoming irrelevant after codebase drift; testable via PR acceptance-rate trends.",
  "reasoning": "Causal claim about churn root cause that's falsifiable. No test path stated, so one is suggested.",
  "statement": "Founder churn at month 2 is caused by Daemon PR proposals becoming less relevant as the codebase shape diverges from initial training conditions.",
  "why_believed": "Founder's pattern-recognition that PR proposals get worse over time correlates with the M2 churn cliff observed elsewhere.",
  "how_will_be_tested": "Track PR-proposal acceptance rate per tenant over time; compare the trajectory for tenants who churn at M2 vs those who survive past M3. (System-suggested test path — not stated by founder.)",
  "confidence": 0.85
}}}}

Rules:
- If confidence < 0.6, return {{"proposal": null}} instead.
- A hypothesis requires a TESTABLE element — vague beliefs don't qualify.
- If the turn describes multiple hypotheses, return only the most prominent one.
- If unsure whether this is a hypothesis vs. a decision or feature, return {{"proposal": null}}.
- Better to miss a real hypothesis than to propose a false one.
